## Mode Detection

Parse `$ARGUMENTS` to determine the mode:

- **Discover mode** if the input matches any of:
  - `surprise me` (any casing)
  - `something in <area>` or `something about <area>`
  - An area/domain name alone with no specific topic (e.g., "databases", "music theory", "quantum computing")
- **Explain mode** if the input is a specific named topic (e.g., "CRDT", "htmx", "Zettelkasten", "Bun")

If the input is ambiguous — it could be either a specific topic or a broad area — ask the user: "Did you mean you want me to explain **[topic]**, or discover something interesting in the **[area]** space?"

## Explain Mode

### Step 1: Disambiguate

If the topic name is ambiguous (multiple well-known meanings), ask the user which one before proceeding. Examples:
- "Mercury" → planet, programming language, or element?
- "Spring" → Java framework, the season, or Spring (the startup)?
- "Redux" → React state management or the general concept?

If the topic is clear, proceed directly.

### Step 2: Check for Existing Page

Check `docs/whatsit/` for an existing file matching the topic (use Glob). If one exists, tell the user: "I already have a page on **[topic]** at `docs/whatsit/[file]`. Want me to update it instead?"

### Step 3: Research

Use **WebSearch** to gather information. Run multiple searches to cover:

1. Official site/repo and documentation
2. "What is [topic]" — overview and problem statement
3. "[topic] vs alternatives" or "[topic] compared to" — competitive landscape
4. "[topic] getting started" or "[topic] tutorial" — onboarding experience
5. "[topic] criticism" or "[topic] drawbacks" — honest weaknesses
6. "[topic] production use" or "[topic] who uses" — maturity signals

Extract **real URLs** from search results. Every source cited must be a real URL you found, never fabricated.

## Discover Mode

### Step 1: Determine the Area

- If `surprise me`: pick any domain — tech, science, history, philosophy, linguistics, mathematics, whatever seems interesting. Vary across invocations.
- If `something in <area>`: use that area.
- If just a broad area name: use that.

### Step 2: Check Existing Pages

Read the list of files in `docs/whatsit/` to know what's already been covered. **Never pitch something that already has a page.**

### Step 3: Hunt for Something Interesting

Use **WebSearch** to find something the user probably hasn't heard of. Search strategies:

- For tech/SDE areas: search for "underrated [area] tools", "hidden gem [area] library", "new [area] project 2025 2026", "[area] tool nobody talks about", "awesome [area] list"
- For non-tech areas: search for "obscure [area] facts", "counterintuitive [area]", "most interesting [area] research", "weird [area]"

**Lane selection (tech/SDE only):**
- **[Proven]** — Battle-tested, real users, real production use, but somehow under the radar. The vibe is "how did I not know about this?"
- **[Emerging]** — New, buzzy, possibly polarizing. Real excitement even if the jury's still out.

Skip anything that's mid-tier obscure with no community signal (no stars, no users, no discussion). The goal is either genuinely useful or genuinely fascinating.

For non-tech domains, optimize for **interestingness** — obscure, academic, counterintuitive, and weird are all welcome. No lane labels needed.

### Step 4: Pitch (Do NOT Create the Page Yet)

Present a **~50-word pitch** to the user in this format:

> **[Lane if tech] [Topic Name]** — [50-word pitch explaining what it is and why it's interesting]
>
> Say **yes** to get the full page, or **"show me another"** to keep browsing.

**Wait for the user's response.** Do not create the page until they say yes.

### Step 5: On "Yes" — Create the Page

Follow Explain Mode Steps 3–5 (Research → Write → Confirm).

### Step 6: On "Show Me Another"

Return to Step 3 and find a different topic. Don't repeat previous pitches from this session.

---

## Important Notes

- **Real URLs only.** Every source must come from an actual WebSearch result. Never invent URLs.
- **Scannable format.** Bullets and tables over prose. The page is a reference, not an essay.
- **80–150 lines.** Enough to be useful, short enough to scan in 2 minutes.
- **No hype, no hedge.** The Honest Assessment section should be genuinely honest. If something is immature, say so. If it's great, say so.
- **Topic slugs:** lowercase, hyphens, no special chars (e.g., `htmx.md`, `conflict-free-replicated-data-types.md`, `zettelkasten.md`).
- **Non-tech is welcome.** Whatsit covers anything — programming tools, math concepts, historical events, cognitive biases, musical techniques, whatever the user asks about.
