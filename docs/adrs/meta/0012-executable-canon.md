# ADR-0012: Executable Canon — Enforce Methodology via Harness Primitives, Not a Standalone Control Plane

## Status

Proposed (2026-07-24) — awaiting human ratification. This ADR is the **direction, not the build**: it decides *where* Summon's next-generation enforcement lives (in this repo, on the agent harness's own primitives) and *what is rejected* (a standalone external supervisor). The individual translations are implementation work items sequenced after ratification per the Sequencing section.

**Gate record.** Architecture Gate completed 2026-07-24. Archie reviewed (14 findings: 2 Critical, 8 Important, 4 Minor — both Criticals were placement/composition defects in the first draft, repaired in this revision). Wei challenged (8 objections: 6 demanded amendments, all incorporated; 1 mechanical fix, applied; the persona-retirement attack was argued and **withdrawn** conditional on the revisit trigger now in sub-decision F). Point-by-point dispositions: `docs/history/tracking/2026-07-24-adr-0012-executable-canon-debate.md`. This ADR is **meta** by ADR-0007 §1's subject test (it is about building Summon) and lives in `docs/adrs/meta/` accordingly — the first draft sat in canon, which Archie and Wei each independently flagged as a violation of the boundary ADR it cites.

## Context

**The provocation.** `docs/history/codex_rewrite_design_doc.md` (moved to the history zone with this ADR — it is third-party reference material, not canon, and must not scaffold into user projects) is a 2,609-line design for rebuilding Summon as a standalone Python control plane. Provenance matters for weighing it: it was authored by **OpenAI's Codex CLI** on 2026-07-15 — not by this project's agents — in answer to a prompt about *Codex's* tier differences, and it targets Codex's SDK, sandbox, and protocol surface throughout. It is a serious, well-researched document, and this ADR treats it as a hostile-friendly external review: its diagnosis is accepted, its prescription is not.

**The diagnosis is verified.** The doc's §5.4 limitations were checked against this repo on 2026-07-24 and hold up:

1. **Prompt-only gates.** The same coordinator that does the work must remember to invoke its own checks. The Done Gate, the sentinel-verification ritual, and the review wave are all prose that context pressure can silently drop.
2. **Self-authored completion evidence.** A worker's sentinel and report are produced by the process they are meant to validate — CLAUDE.md's "Treat Agent Output as Untrusted" rule exists precisely because this fails.
3. **Capability drift.** Declared, implemented, and advertised diverge: `summon-team add <runtime>` is advertised in the installer (`packages/summon-team/src/index.ts:148`) while the ADR-0006 generator is still gated; the executable doctor, the `/doctor` command, and the canon checker cover different invariants.

**The prescription doesn't transfer — with the claim stated honestly.** The doc's foundational decision D1 — *build a deterministic external supervisor, because the model will not reliably self-orchestrate and scheduling must be a program decision* — is a premise about Codex's runtime. On the Claude Code harness, the equivalent primitives exist as shipped features: **hooks** (deterministic blocking of tool calls), **worktree isolation** per agent, **structured outputs** validated at the tool layer, and **Workflow scripts** (deterministic control flow with fan-out, schema-validated agent outputs, budgets, resumable journals). One caveat is load-bearing and Wei is on record demanding it: Workflow is a **research preview** (shipped 2026-05-28, plan-gated — unavailable on some plan tiers of the same runtime). D1 is therefore *conditionally* answered, not falsified: hooks and worktrees are stable substrate; the ceremony layer bets on a preview feature and is earn-gated accordingly (sub-decision C). This ADR keeps the codex doc's version discipline even while rejecting its architecture: every enforcement asset declares its required primitive and minimum harness version in the capability registry (E), and `doctor` probes availability rather than assuming it.

**The model shift.** Summon's methodology was written against Opus-class coordinators, which needed heavy prose scaffolding and still dropped steps under context pressure. Fable-class models follow long process docs more reliably *and* sit in a harness whose primitives can enforce the steps that matter regardless of model discipline. The correct response to a stronger model is not a bigger external cage; it is moving each check to the cheapest layer that makes it deterministic.

**The constraint stack.** Two ratified ADRs bound this decision. ADR-0007 (canon/meta boundary) governs what ships into scaffolded projects versus what serves only this repo. ADR-0006 (multi-runtime install) commits Summon to a Copilot consumption tier — **committed, gated behind a fidelity proof and real user demand, and second-tier by design**, not first-class. Any enforcement layer built on harness primitives must degrade explicitly, not silently, wherever a primitive is missing — and per Wei, "wherever" includes plan tiers within Claude Code, not only other runtimes.

## Decision

Summon's next generation is an **in-repo translation of its methodology from prose into executable harness primitives**. No standalone control plane, no separate repository, no persona retirement. Six sub-decisions:

**A — Direction: evolve this repo; reject the standalone supervisor.** The codex doc's control plane is rejected *as a Summon direction*: it is a platform-security product for a different audience (its v1 includes event-sourced state, cgroup kill domains, and an SSRF egress gateway), and it rebuilds what the harness already provides. It is recorded in Alternatives as a legitimate *separate product* someone could build; Summon is not it.

**B — The enforcement ladder.** Every methodology rule gets classified by the cheapest layer that makes it deterministic:

| Layer | What lives here | Examples |
|---|---|---|
| **Hook** (code blocks the action) | Checks that require **moment-of-action blocking** | The **exit-path hook**: commit/push blocked absent required ceremony receipts for the current work item; commit blocked on missing agent-notes update |
| **Script** (code verifies on demand) | Checks decidable from repo state, run by doctor/CI | Done-gate deterministic items (tests ran, board item exists), canon checker, debt harvester, capability-registry check (E) |
| **Workflow** (deterministic orchestration) | Multi-agent ceremonies whose *structure* must not be skippable | Review wave, architecture debate, sprint-boundary sweep, Parallel Work dispatch |
| **Prose** (judgment) | Genuinely inferential or human calls | Proof-grade assignment for inferential claims, risk appetite, taste |

Two rules govern placement. **Tie-breaker (Wei):** when a script achieves the same determinism as a hook, prefer the script — it runs on every runtime and plan tier; hooks are reserved for checks that must block at the moment of action. **Policy amendment (named explicitly, per Archie):** the ratified encoding norm — done-gate § "Encode the fix, not the memory," which encodes a sensor *when an item keeps failing the same way* — is hereby amended for **deterministic** checks only: a deterministically-checkable rule left in prose after its layer is available is a defect, without waiting for an observed failure, because the Fable-era marginal cost of a sensor has dropped below the cost of the failure it waits for. Observed-failure remains the bar for *inferential* sensors. This is a deliberate change to ratified policy, on the record here, and its cost (an open-ended sensor backlog with fixtures to maintain) is priced in Consequences.

The proof-grade ladder maps directly: **deterministic** claims belong in hooks or scripts; **inferential** claims stay with agents but their *outputs* become schema-constrained; **human** claims stay human.

**Receipts (deferred design, owned by Archie).** "Receipts over claims" — the codex doc's strongest principle (its D8: evidence is bound to an exact tree state and invalidated when the tree changes) — is adopted but not designed here. The receipt schema (who produces the evidence artifact, what tree state it binds to, what invalidates it) is an explicit follow-up design item in E's registry scope, owned by Archie, sequenced before the exit-path hook that consumes receipts. Until it lands, one rule holds: a hook or script that verifies a deterministic claim **re-executes the check itself** (or verifies a fresh artifact it can bind to current state) rather than trusting a worker-reported result — a format-valid sentinel is still self-authored evidence, and format checks close only half of §5.4(2).

**C — Ceremonies become Workflow scripts, earn-gated.** The end state: checked-in Workflow scripts with fixed fan-out, schema-constrained findings, adversarial verification of findings before they reach the human, and a roster the coordinator cannot "forget." Because Workflow is a research preview, this sub-decision is earn-gated on the ADR-0006 pattern ("earn the generator before building it"): the **review wave alone** is built first, with at least one negative-control fixture, and must survive real sessions before the architecture-gate, sprint-boundary, and Parallel Work dispatch workflows are sequenced. The review-wave roster is the **governance-defined one, computed from the diff by the script** — Vik + Tara + Pierrot + Archie's conformance lens, plus trigger-based conditional lenses (Dani on any UI change, migration-safety and API-compat when relevant) — not a hard-coded three-lens list; a fixed wrong roster would convert today's possible omission into a structural one. Rosters take additions at invocation; omissions are deliberate. Slash commands stay as the human-facing entry points and invoke the scripts, and each ceremony **announces its mode at invocation** ("review wave: prose mode — Workflows unavailable on this install") per the portability clause. Known prerequisite (Archie M2): team-governance still cites the pre-0007 debate-artifact path; that path fix sequences ahead of the gate workflow so the script doesn't freeze a stale path.

**D — Parallel mutating agents get worktree isolation — enforced where a workflow spawns them.** Each mutating agent in a scripted ceremony runs under `isolation: worktree`. Stated honestly per Archie: this is *structurally* enforced only once the Parallel Work dispatch workflow (C's fourth ceremony) exists; ad-hoc parallelism remains prose-guarded until then, which is why D sequences strictly after its carrier ceremony.

**E — The doctor grows a capability registry, keyed on probed capabilities.** One machine-readable registry of what Summon advertises (commands, gates, ceremonies, scaffold outputs), each entry carrying: implemented/gated/planned status, required harness primitive and minimum version, and enforcement level. Per Wei's sharpest demand, enforcement level is recorded **per probed capability, not per runtime identity** — `doctor` probes what this install can actually run (hooks? Workflow? worktrees?), because plan tiers split capability within a single runtime and an adapter README nobody reads at invocation time is a disclaimer, not a sensor. Relationship to ADR-0006's manifest (per Archie C2): the **registry is source-of-truth for what exists and what it requires; the manifest records what is installed; doctor joins them** to answer "what is actually enforced here?" Registry ownership: Archie owns the schema (versioned with a discriminator, validated, on the manifest's discipline); the doctor's maintainer owns the probe checks. The registry file's zone: the check script is canon (scaffolded projects get their own instance); Summon's own registry data is meta. The `summon-team add` advertisement-versus-reality gap is the first test fixture.

**F — Personas stay; capabilities underneath them tighten.** The codex doc's "roles are capabilities, not characters" (§19.1) is adopted for the *authority* layer — agent definitions keep narrow tool lists, review agents stay read-only — and rejected for the *product* layer. Summon's personas are its pedagogy, its documentation voice, and its brand; they cost context, and that cost is accepted as a product decision, on the record here. **Revisit trigger (Wei's condition for withdrawing the retirement objection):** if ADR-0006's Phase-0 fidelity mapping shows persona frontmatter is the dominant projection-fidelity loss, or measured persona overhead exceeds 10% of a typical session's context budget, F gets re-argued rather than grandfathered.

### Portability and asset-class clause (load-bearing)

Two vocabulary terms, because "canon stays Markdown" and "enforcement assets are canon" contradicted each other in the first draft (Archie I8):

- **Methodology canon** — the Markdown process documents. Portable, runtime-independent, the floor everywhere: on any install lacking a primitive, the methodology remains runnable by prose discipline exactly as today.
- **Enforcement adapters** — executable assets (hooks, scripts, workflow files) that make canon rules deterministic on a given install. **This ADR introduces this asset class; it is new.** It sits *outside* ADR-0006's projection model — 0006's premise is that `.claude/*` Markdown is the single source and other runtimes are derived projections, but a hook script has no derivable Copilot equivalent. Enforcement adapters are therefore per-runtime *originals*, and the drift risk that implies (the exact failure mode 0006's Alternatives rejected) is accepted and mitigated by the registry: every adapter entry names its canon source rule, and the canon checker gains a rule that an enforcement asset without a canon source is a defect.

Each enforcement adapter is classified canon or meta **individually** per ADR-0007 §1. First classifications, so implementation PRs don't re-litigate the boundary: the done-gate check script and capability-registry check — **canon** (they guard user projects; shipping them is the new trust surface priced in Consequences and gated on Pierrot's threat model); the review-wave workflow — **canon**, after the earn gate; the exit-path hook — **canon**, last to ship (it depends on the receipt schema and is the most invasive asset); Summon's own registry data and this ADR — **meta**. Degradation is announced at ceremony invocation (see C), never silently, and never as a claim that a check is deterministic on an install where it is prose.

### Tamper boundary (honesty clause — do not soften downstream)

The codex doc's deepest security property — its D6/D8: integration authority and completion evidence live in a process **outside the model's write authority** — is one Summon **does not get and should not claim**. Our hooks, settings, and workflow scripts live in the same repository the coordinator can write to, and the coordinator invoking ceremonies is itself a model context. This enforcement layer hardens against **drift and forgetting** — the observed failure mode of prose process under context pressure — not against an adversarial or prompt-injected coordinator, which can in principle edit the hooks it is subject to. Summon's actual integration authority is, and remains, **the human reviewing the PR**. The exit-path hook narrows the honest gap (a coordinator that merely *forgets* cannot commit without receipts); it does not close the adversarial one. Downstream docs and marketing must not describe this layer as tamper-proof, supervisor-grade, or equivalent to the control-plane property.

### Reversal triggers (90-day checkpoint: 2026-10-24)

Per Wei: the rejected doc carried reversal criteria (§27); the ADR rejecting it must too. At the checkpoint, or earlier if tripped:

1. **Self-refutation:** the capability registry (E) — the instrument every other claim is measured with — has not shipped. The ADR fails by its own §5.5 logic; stop and rescope.
2. **Ossification beats forgetting:** the review-wave workflow was built and then bypassed in ≥2 real sessions because adapting it cost more than the omission it prevents. Re-argue C's scope.
3. **Preview instability:** ≥2 harness research-preview breakages caused enforcement downtime. Re-weight the ladder toward the script layer.
4. **Two-layer drift:** prose/code drift incidents in enforced rules exceed the prose-only drift rate they replaced. The translation is adding drift, not removing it; revisit B's policy amendment.

### Sequencing

**E first** (the registry is the sensor for everything else and needs no preview primitive), then B's script layer, then B's hooks (exit-path hook last, after the receipt schema), then C's review-wave earn gate, then the remaining ceremonies, with **D strictly after** the Parallel Work dispatch workflow exists. Pierrot's threat-model pass on the shipped-adapter surface (routed per `docs/process/security-intake.md`) gates the first *canon* enforcement adapter, whichever it is.

## Alternatives Considered

- **A. Status quo — prose-enforced methodology.** Rejected: the drift evidence is this repo's own history. Every limitation in the codex doc's §5.4 was verified real. A methodology that depends on the coordinator remembering its own checks degrades exactly when it matters — under context pressure at the end of long sessions.
- **B. The standalone control plane (the codex doc's prescription).** Rejected for Summon: wrong audience (solo devs don't run supervisor daemons), wrong cost (a multi-quarter platform build — no daemon, no event store, no sandbox engineering is required for what Summon needs), wrong substrate premise (D1's supervisor already exists as harness primitives, with the preview caveat stated in Context), and high obsolescence risk — it competes with what both vendors' harness teams ship natively each quarter. Its diagnosis, proof-ladder preservation, receipts-over-claims principle, and version discipline are adopted; the rejection is of the architecture, not the analysis. Its D6/D8 tamper property is explicitly *not* claimable in-repo — see the tamper boundary clause — and anyone who needs that property genuinely needs an external supervisor; that product remains open to build, and it is not Summon.
- **C. Hooks only — skip the Workflow layer.** Considered as the minimal version, and post-gate it is the fallback posture if reversal trigger 3 fires. Rejected as the target: hooks can verify *artifacts* but no hook fires on the *absence* of an agent invocation — a coordinator can still forget a review lens entirely. The Workflow layer is what moves ceremony structure from memory to code.
- **D. Adopt native agent-teams / shared-task coordination as the substrate.** Rejected for now: Summon's ceremonies need deterministic rosters and schema-constrained outputs, not emergent coordination; the codex doc's §26.4 critique of session-led coordination is accepted. Revisit if the primitive matures into something scriptable.
- **E. Retire personas for capability-named roles.** Argued at full strength by Wei at the gate (maintenance burden across 16 files, de-AI pass cost, Copilot projection-fidelity loss, Fable-class models not needing characters to follow process) and **withdrawn** against sub-decision F's split of authority from product, conditional on F's revisit trigger. Recorded with the trigger so the decision is falsifiable, not grandfathered.

## Consequences

### Positive

- Deterministic checks become code: the Done Gate's mechanical items, sentinel verification, and ceremony rosters stop depending on coordinator discipline — the failure mode behind most of this repo's recorded process violations.
- The review wave and architecture gate produce schema-valid findings instead of free-text reports, making the "treat agent output as untrusted" rule enforceable rather than aspirational — with the exit-path hook giving the ladder a floor: forgetting a ceremony can no longer end in a merged commit.
- Capability drift gets a sensor (the registry) instead of periodic audits that find it months late — and the same registry carries version pins and probe requirements, converting vendor exposure from accepted risk to sensed risk.
- The work decomposes into independently shippable, explicitly sequenced items rather than a big-bang rewrite; each lands through the normal board/PR process.
- The codex doc's genuinely good ideas (proof ladder as typed layers, receipts over claims, negative-control fixtures, version discipline) arrive without a daemon, an event store, or sandbox engineering.

### Negative

- **Two-layer maintenance.** Every enforced rule now exists as prose (canon) *and* code (adapter); they can drift from each other. Mitigation: registry entries name their canon source and the canon checker enforces the linkage — but the burden is real and permanent, and reversal trigger 4 watches whether it exceeds the drift it replaced.
- **A new trust surface: Summon ships executable code.** Today `npx summon-team` installs Markdown; under this ADR, canon enforcement adapters execute on user machines — on every matching tool call, in the case of hooks. That is a categorical posture change: ADRs 0010/0011 hardened Summon's *inbound* supply chain; nothing yet governs Summon *as* a dependency. Pierrot's threat-model pass gates the first canon adapter, every hook ships with a documented per-hook escape hatch, and a false-positive policy is required before any blocking hook lands — a hard-fail false positive in a stranger's repo is a product-breaking bug, and the rational user response (rip the hooks out, silently reverting to prose while marketing still says "deterministic") is priced here, not discovered later.
- **Vendor-primitive exposure, sharpened.** The ceremony layer bets on a plan-gated research preview. Mitigations: the earn gate on C, per-asset version pins in the registry, doctor probes, prose as the universal floor, and reversal trigger 3. Accepted with eyes open, not "fails loud" hand-waving.
- **Unequal installs, honestly labeled.** Per the portability clause, installs lacking a primitive get weaker enforcement, announced at invocation. Summon's marketing must not claim deterministic gates on installs where the gate is prose.
- **The policy amendment creates a backlog.** B's shift from encode-on-observed-failure to encode-when-deterministic means an open-ended queue of sensors, each with negative-control fixtures to maintain forever, and each blocking hook carrying a per-session latency tax that belongs in its acceptance criteria.
- **Ceremony scripts can ossify.** A checked-in Workflow is harder to adapt mid-session than prose. Mitigation: rosters computed from the diff with additions accepted at invocation; reversal trigger 2 watches the bypass rate.

### Neutral

- The codex doc lives at `docs/history/codex_rewrite_design_doc.md` — third-party reference material in the non-shipping zone, preserved as this ADR's analyzed input.
- Nothing here changes the phase model, the persona catalog, or the tracking workflow; this ADR relocates *enforcement*, not *methodology*.
- The standalone-product path (Alternative B) remains open to anyone — including a future Summon spin-off — as a separate venture with its own cost-benefit; this ADR only decides it is not Summon.
