# Code Review: `summon doctor` v1 — portable health registry + CLI dispatch

**Date:** 2026-06-24
**Reviewed by:** Vik (simplicity), Tara (testing), Pierrot (security), Archie (conformance), Ines (operational)
**Files reviewed:** `packages/summon-team/src/doctor.ts` (new), `packages/summon-team/src/index.ts` (modified), `packages/summon-team/test/doctor.test.ts` (new)
**Verdict:** Approved with suggestions — two Important findings (test/build coupling; multi-block regex fragility), no Critical.

## Context

First implementation increment of `summon doctor` (issue #30, descoped per Accepted ADR-0004). Adds a second invocation mode to the `summon-team` CLI: `summon-team doctor` runs a portable `health` registry against the current working directory, prints a human summary, and exits 0 (healthy) / 1 (a check errored). It downloads nothing, writes nothing, touches no git.

The v1 registry has exactly one check — `checkAgentNotesDeps` — which verifies that agent-notes `deps:` paths under `.claude/` and `docs/` resolve to real files, with two precision rules learned from dogfooding (skip trailing-slash directory deps; only flag a missing file when its parent directory exists). The `canon` registry (framework self-consistency) is deliberately excluded from doctor's import graph per ADR-0004 Decision 2.

I verified the green claims independently: `npx tsup` builds clean; `npx vitest run test/doctor.test.ts` is 11/11 green; `node dist/index.js doctor` at repo root prints `healthy — 1 check(s) passed` and exits 0. The compiled bundle (`dist/index.js`) has **zero** functional `canon` references — the structural-unreachability contract from ADR-0004 holds.

## Findings

### Critical

None. No security surface (Tier-0/1, read-only, no network, no writes, no user-input-to-shell path). Pierrot: clean — see Lessons. No data-loss or broken-functionality risks.

### Important

**1. Integration tests silently require a prior build; `test` has no `pretest` hook.**
`test/doctor.test.ts` lines 13/117–134 spawn `node dist/index.js`. But `dist/` is gitignored and `package.json`'s `test` script is bare `vitest run` with no `pretest: tsup`. I confirmed empirically: with `dist/` removed, **3 of 11 tests fail** (`MODULE_NOT_FOUND`) — the 3 CLI-dispatch tests. On a clean checkout or in CI without a preceding `build` step, `pnpm test` is red. Worse than red: if a *stale* `dist/` is present, the CLI tests pass against old code and silently mask a regression in `index.ts`'s dispatch.
*Why it matters:* a test suite that depends on an un-versioned, separately-built artifact, with no enforced ordering, is a flaky-green/false-green trap — exactly the failure mode the project's "treat agent output as untrusted" rule exists to catch, but at the CI layer.
*Fix:* add `"pretest": "tsup"` (or `"test": "tsup && vitest run"`) to `packages/summon-team/package.json`. Optionally have the integration `describe` block fail loudly with a "run build first" message if `CLI` doesn't exist, so the failure is diagnostic rather than a raw module-not-found stack.

**2. `extractDeps` matches only the FIRST `deps:` block per file (no `/g`); correct today only by corpus coincidence.**
`doctor.ts:50` uses `text.match(/deps:\s*\[([^\]]*)\]/)` — a single, non-global match. Files in this repo legitimately contain multiple `deps:` occurrences (e.g. `docs/process/tracking-protocol.md`, `.claude/commands/cloud-update.md`, `docs/methodology/agent-notes.md` has 13). I classified them: the later occurrences are all either prose ("optional deps:") or **embedded agent-notes templates** inside fenced examples. So today the regex checks the file's own real first-line agent-notes and ignores the examples — which happens to be the *right* behaviour.
*Why it matters:* this correctness is accidental, not designed. It rests on the unstated invariant "the file's real agent-notes is always the first `deps:` in the file." A future doc that opens with an example block, or that legitimately carries content the author wants doctor to verify in a second block, will be silently skipped — a false-negative with no test guarding against it. The naive reader of `doctor.ts` cannot tell the single-match is intentional.
*Fix:* either (a) document the "first block only, by design — agent-notes is always file-leading" invariant in a comment next to the regex and add a test pinning it, or (b) anchor the match to the file's leading agent-notes block explicitly (the spec says agent-notes is the first metadata line/comment). Don't switch to global match blindly — that would start verifying example-template deps and reintroduce false-positives.

### Suggestions

**3. YAML multi-line array form yields `[]` silently.** I probed `extractDeps` across forms: inline `[a,b]`, quoted, empty, YAML-frontmatter-block `deps: [a, b]`, and deps spanning a newline *inside* brackets all parse correctly. Only the YAML *list* form (`deps:\n  - a\n  - b`) returns `[]`. The agent-notes spec (`docs/methodology/agent-notes.md`) canonicalizes the inline `deps: [...]` form exclusively — every example uses it — so this is not a live false-negative against the canonical corpus. Worth a one-line comment on `extractDeps` noting it assumes the bracket form, so a future spec change that permits the list form doesn't silently under-check.

**4. Nested-subdir false-negative is slightly wider than the documented rule implies.** The parent-dir-exists rule is sound and the trade-off is correct (the naive version produced 9 false-positives; this reduces to the 1 genuine bug). I constructed the boundary case: a dep `docs/a/b/c.md` where `docs/a/` exists but `docs/a/b/` does not returns `ok` — the suppression keys on `dirname(dep)`, so a typo in a *subdirectory* of a real category still escapes, not just an absent top-level category. This is consistent with the design intent (absent dir = not-yet-generated) and I'd keep it, but the code comment at lines 81–83 frames it as "absent *whole category*," which undersells the depth at which suppression kicks in. Tighten the comment.

**5. `isSummonProject` = `.claude/` exists is intentionally weak, and that's fine.** I probed whether doctor could run somewhere harmful: it cannot. It is read-only (no writes, no network, no git, no shell-out — confirmed by tracing `runDoctor` in `index.ts:41-52` and the import surface of `doctor.ts`: only `node:fs` reads + `node:path`). The worst case of a false-positive `.claude/` match is doctor walking some unrelated `.claude/` + `docs/` tree and printing a result — no side effect. Marginally stronger gates (require `CLAUDE.md` too) would reduce confusing output but buy no safety. Leave as-is; the weak gate is proportional to a read-only diagnostic.

**6. `formatResults` never emits the `degraded` icon path in v1.** `doctor.ts:135` handles `degraded` (`!`), but no v1 check returns it and `exitCodeFor` only branches on `error`. This is fine — it's the shared struct from ADR-0005 — but it's untested dead-ish code today. A one-line unit test that feeds a hand-built `degraded` result through `formatResults`/`exitCodeFor` would pin the intended semantics (degraded → exit 0, `!` icon) before a real check relies on them.

### Clean

- **Pierrot (security/compliance):** No security or compliance concerns. The diff adds a read-only diagnostic with no user-input-to-sink path, no secrets, no new dependencies, no new network or attack surface. `runDoctor` reads files and prints; it shells out to nothing. The `doctor` branch sits *before* all scaffolding logic in `main()` (`index.ts:57-60`), so the download/git-init/file-delete machinery is provably unreachable in doctor mode.
- **Archie (architectural conformance):** The canon/health boundary from ADR-0004 is upheld at the strongest level — structural unreachability. `doctor.ts` imports nothing canon-related (only `node:fs`/`node:path`), and the compiled `dist/index.js` contains zero functional canon references. The internal `CheckResult` struct matches ADR-0004 Decision 3 (`verdict ∈ ok|degraded|error`, `evidence[]`, `schemaVersion`) and is correctly *not* serialized — exit-code + stdout is the only contract crossing the boundary, as specified. No consumer-specific concept leaks into the shared struct.
- **Ines (operational):** Failure modes are surfaced appropriately for a developer-invoked CLI: non-Summon-dir prints an actionable message to stderr and exits 1; broken deps name the offending `file -> dep` pairs in stdout. No logging framework is warranted for a one-shot diagnostic. The exit-code contract is the operational signal and it is correct.

## Lessons

1. **A test that spawns a build artifact must own the build.** Tests reaching into `dist/` are a standard integration pattern, but the build–test ordering has to be *encoded* (`pretest`), not assumed. An un-ordered dependency on a gitignored artifact has two failure modes that look opposite: red on a clean checkout (missing artifact) and false-green on a dirty one (stale artifact). The stale-green is the dangerous one — it lets a dispatch regression ship while CI stays green. When you see `node dist/...` or `import '../dist/...'` in a test, immediately ask: "what guarantees this is freshly built?"

2. **"Correct by coincidence" is a latent bug.** The single-`match` regex does the right thing here *only because* this repo's real agent-notes always lead the file and later `deps:` are examples. That's an invariant nobody wrote down and no test pins. The lesson: when code is correct because of a property of the current data rather than a property the code enforces, either make the code enforce it or pin the property with a test and a comment. Otherwise the next data shape that violates the unstated invariant fails silently — and silent under-checking in a *health check* is the worst place for it, because the tool's whole job is to not miss things.

3. **Designed false-negatives are a legitimate, even admirable, choice — but state their true reach.** The parent-dir-exists rule deliberately trades "miss a typo in an absent category" for "never false-FAIL a healthy project," and that trade is correct: a check that false-FAILs a correct project is as harmful as one that never fires (ADR-0004 Decision 4 says exactly this, and the 10→1 dogfooding result proves it). The only nit is precision of the *comment*: the suppression keys on `dirname(dep)` at full depth, so it's slightly broader than "whole category." When you encode a deliberate blind spot, describe its exact edge, because the comment is the only place a future maintainer learns it was a choice and not an oversight.

4. **Structural unreachability beats runtime guards for "this must never ship downstream."** ADR-0004 could have kept canon out of `doctor` with a `cwd`-sniffing runtime check ("am I in the Summon repo?"). Instead it made canon physically absent from the compiled entrypoint's import graph — and that's verifiable with a single `grep` on the bundle, which I ran. The lesson: when a boundary must hold, prefer one the build *enforces* (and a reviewer can mechanically check) over one that depends on a runtime guess that can leak or misfire. "I checked the compiled artifact, not just the source" is a habit worth keeping for any "X never ships" claim.
